これは何か

Overview

数式のスクリーンショットを貼ると、Gemini の画像認識で中身を読み取り、LaTeX テキストにして返す。 出したテキストはそのまま ChatGPT に「文字」で送れるし、区切り記号を外して Word の数式にも貼れる。

きっかけは単純で、ChatGPT への画像貼り付けには1日の厳しい制限があるのに、テキストのやり取りには実質制限がないこと。 だったら「画像 → 文字」の一手間を自動化すれば、制限の壁ごと消える。

設計思想

Design Philosophy

数式は「読める・理解できる」のに「入力が異常にだるい」典型だ。\frac や上下付き、総和記号を手で打ち直す時間が、 学習やレポート作成のたびに積み上がる。SnapTeX はその入力工程だけを丸ごと外すことに機能を寄せた。

Before
  1. 数式を見る
  2. 頭でLaTeX/Word数式に翻訳
  3. 手で打ち直す（一番遅い）
  4. ミスを直す
After — SnapTeX
  1. スクショを撮る
  2. 貼る（PCはCtrl+V／スマホはタップ選択）
  3. 変換
  4. コピーして貼る

「賢い一発回答」を目指すのではなく、汎用（ChatGPTへ）と専用（Wordへ）の両方を1つの道具で賄い、 毎日の摩擦を小さくすることに価値を置いている。

主要な技術的判断

Key Decisions — 課題 / 判断 / 理由

Wordに貼ると $$ が残る問題

課題
Wordの数式は $…$ / $$…$$ という区切りを解釈せず、文字として残ってしまう。
判断
区切り記号トグルを用意し、「$付き（ChatGPT向け）」と「なし（Word向け）」を生成しなおさず出力だけ切替。中身の \frac 等は保持したまま $ だけ剥がす。
理由
認識は1回で十分。用途（送り先）によって表層だけ変えれば、APIコールも待ち時間も増えない。

複数式を一度に扱う

課題
1枚に何式もある教科書・論文ページ。Wordは「1式＝1数式ボックス」が一番きれいに組める。
判断
式ごとコピー——出力を $$…$$ 単位で切り出し、式ごとにプレビュー＋コピーボタンを並べる。Alt+= のボックスへ1つずつ確実に貼れる。
理由
「まとめて1回」より「1式ずつ手応えを持って」の方が、貼り先を選べて事故らない。手動コピーの確認感はあえて残した。

無料枠のレート制限に耐える

課題
Gemini無料枠には上限（429）や一時的な混雑（503）がある。
判断
上限に当たったら自動で別モデルへ巡回（無料枠はモデルごとに別枠）、混雑は待って自動リトライ、エラーは日本語で「いつ直るか」を案内。
理由
個人利用でサーバーを持たない前提なら、詰まりどころを先回りで吸収しておくのが一番効く。

鍵はコードに書かない（BYOK）

課題
静的配信でAPIを使うと、鍵の置き場所が問題になる。
判断
各自の鍵を設定画面で入力し localStorage にのみ保存。サーバーを介さず、Google以外へは送らない。
理由
ノーサーバーを保ったまま安全側に倒す。共有PCでは設定から即消せる。

制約と割り切り

Honest Limitations

Trade-offs
  • 認識精度はモデル依存。 複雑な多段分数・手書き・低解像度・化学構造などは誤ることがある。だからプレビューで目視確認 → テキストを直接手直しできる導線を必ず残した。
  • プレビューはネット接続時のみ（MathJaxをCDN取得）。オフラインでも変換テキスト自体は使える。
  • 画像はGemini（Google）へ送信される。 アプリ側には保存しないが、機密資料の扱いは利用者の判断に委ねる。
  • 混在モードの「区切りなし」はインライン数式の目印が消える。 Word用途では「数式だけ」モードを推奨、という設計上の割り切り。
